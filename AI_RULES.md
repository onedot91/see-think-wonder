# AI_RULES.md

## 1. 프로젝트 목적

이 프로젝트는 초등학교 3학년 학생들이 수업 자료를 보고, 생각하고, 궁금한 점을 차례대로 정리하는 웹사이트이다.

핵심은 `본 것 -> 생각한 것 -> 궁금한 것`의 흐름이 끊기지 않게 돕는 것이다. 학생은 각자 크롬북에서 작성하고, 교사는 교사 PC에서 모둠별 결과를 모아 볼 수 있어야 한다.

결과는 브라우저 한 대에만 저장하지 않고 Supabase에 저장하여 여러 기기에서 함께 확인할 수 있게 한다.

## 2. 사용자

주 사용자는 초등학교 3학년 학생과 초등학교 교사이다.

학생은 긴 설명 없이 바로 누르고 쓸 수 있어야 한다. 교사는 프로그래밍 비전공자일 수 있으므로 실행, 설정, 수정 방법이 단순해야 한다.

## 3. 디자인 원칙

- 화면에는 꼭 필요한 문구만 둔다.
- 글자는 교실 뒤에서도 보일 만큼 크게 둔다.
- 버튼은 크고 분명하게 만든다.
- 학생 입력 화면은 `본 것`, `생각한 것`, `궁금한 것`의 순서를 강제한다.
- 세 단계는 색으로 구분하되 너무 복잡하게 꾸미지 않는다.
- 입력 전에는 빈칸이 분명히 보이고, 입력 후에는 완료된 느낌이 나야 한다.
- 같은 역할을 하는 카드, 버튼, 입력칸은 같은 모양과 크기 규칙을 유지한다.
- 애니메이션은 단계가 이어진다는 느낌을 줄 때만 사용하고, 과하게 움직이지 않는다.

## 4. 개발 원칙

- 수업에 바로 필요한 기능을 먼저 만든다.
- 기능을 추가할 때는 학생 화면이 더 복잡해지지 않는지 확인한다.
- 학생이 쓴 내용이 사라지지 않도록 저장 흐름을 조심해서 다룬다.
- 공유 데이터는 Supabase의 `stw_responses` 테이블을 기준으로 읽고 쓴다.
- 교사 화면은 모둠별 결과를 문장틀 형태로 크게 보여주는 것을 우선한다.
- 오류 문구는 짧고 쉬운 한국어로 쓴다.
- 개인정보는 모둠 번호 정도만 사용하고, 학생 이름이나 학번은 기본으로 모으지 않는다.

## 5. 교사가 바이브 코딩할 때 지킬 점

- 한 번에 많은 기능을 바꾸지 않는다.
- 색, 문구, 글자 크기처럼 수업 중 바로 보이는 것부터 작게 수정한다.
- Supabase 주소와 anon key는 `.env`에만 적고 코드에 직접 넣지 않는다.
- 설정 방법은 README에 짧게 남긴다.
- 수정 뒤에는 `npm run build`로 깨지는 부분이 없는지 확인한다.
- 실제 수업 전에 교사 화면과 학생 화면을 각각 한 번씩 열어 제출과 결과 확인을 테스트한다.

## 6. 과하지 않은 규칙

- 복잡한 로그인은 당장 필요할 때만 붙인다.
- 학생 화면에는 설명보다 행동을 우선한다.
- 교사 화면에는 한눈에 공유할 수 있는 결과를 우선한다.
- 기술적으로 멋진 구조보다 수업에서 안정적으로 작동하는 구조를 우선한다.
